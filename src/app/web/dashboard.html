Source des bougies : uniquement l'exchange (CCXT) — pas le CSV workspace. À chaque cycle le serveur télécharge une fenêtre de plusieurs centaines de bougies (glissante, avec la dernière clôture), puis lance l'analyse SMC : tu as déjà l'historique nécessaire pour les indicateurs ; chaque tick rafraîchit la fin de série. Si symbole / TF paper sont vides, la boucle prend la 1re entrée de symbols et timeframes dans ta méthode JSON.

Bitget futures : mets bitget dans « Exchange CCXT » et le backend bitget_futures_sim pour que bougies + frais auto passent par Bitget comme en prod, tout en gardant la simulation replay pour les ordres. Enregistre (bouton run enregistre aussi ces champs) puis relance le run.

Trades paper (simulation, 40 derniers)

Trades fictifs (même moteur que le backtest) : le signal est le top setup sur la dernière bougie ; l'entrée n'a lieu que si une bougie suivante touche le prix d'entrée (comme le replay walk-forward). Les sorties (SL, TP, trailing, TIMEOUT max holding) et les frais / funding suivent l'état Contrôle / Simulation. Ce n'est pas relié à un compte exchange ni au carnet CCXT réel.

Telegram (paper) : avec bot + chat configurés (.env) et paper_live_send_telegram activé dans la méthode / l'état, le serveur n'envoie que les messages liés aux trades simulés (une notification à l'entrée replay, une autre à la clôture) — plus de résumé SMC à chaque poll.

Contexte de simulation

Les listes dataset et méthode sont dans les onglets Dataset et Méthodes ; le serveur enregistre la sélection tout seul. Symbole / timeframe pour le backtest : onglet Dataset. Pas besoin de « charger / sauver un état » ici pour un usage courant.

Tester la méthode (trades simulés)

Choisir symbole / timeframe et source OHLCV (onglet Dataset), puis un fichier méthode JSON (liste workspace). Backtest rejoue la stratégie sur l'historique chargé (positions simulées, frais, rapport de trades). Optimisation explore la grille de paramètres ; le Top-1 peut être écrit dans le JSON méthode si un fichier est sélectionné. Avant chaque run, les champs ci-dessous sont synchronisés avec le serveur. Les blocs Dernière optimisation et Walk-forward indiquent ensuite quelle méthode et quelle source de bougies ont servi au run (symbole / TF = champs Dataset ci-dessus).

File d'optimisations (plusieurs jobs séquentiels)

Même symbole / timeframe que ci-dessus. Chaque job lit une méthode source (JSON workspace), optimise, puis enregistre une nouvelle copie nom__autoopt__…json avec métriques Top‑1 (win rate, PnL, etc.). Le fichier source n'est pas écrasé.

Dataset — symbole, timeframe, historique

Choisis un CSV déjà présent dans la liste, ou crée-en un avec le bloc ci-dessous (une seule action principale). Les dossiers sont indiqués sous la liste après rafraîchissement.

Créer un dataset (recommandé)

Télécharge depuis l'exchange (CCXT) et enregistre un CSV dans data/dashboard_datasets/, puis active ce fichier comme source. C'est le flux le plus simple pour les backtests sur fichier.

≈ — bougies (jours → barres, plafonné 20 000) si la cible ci-dessous est vide.
Laisse vide pour un nom généré (symbole normalisé + tf + jours).
Si rempli, ignore le calcul « jours → bougies » et assemble jusqu'à ce nombre (max 100 000), en enchaînant les pages exchange (~1000/requête).
Ajoute les nouvelles bougies au fichier existant (même basename), dédoublonnage par horodatage.
État : en attente. Renseigne symbole + timeframe, puis clique sur le bouton vert ci-dessus (durée variable selon l'exchange).

Après succès : le CSV devient la source « fichier », les limites backtest/optim sont alignées sur le nombre de bougies reçues, et le résumé apparaît aussi dans « Dernier fetch » plus bas.

Période backtest / optim / WF : réglée dans l'onglet Simulation (presets, plafond bougies, limites personnalisées).
